# Текущая оценка и цели проекта drv60870

**Дата:** 2025-03-14

## Цели проекта

1. **Симулятор полевого уровня телемеханики** — имитация КП (контроллеров/РТУ) по протоколу МЭК 60870-5-104 для проверки SCADA (WinCC OA и аналоги).
2. **Учебный характер** — получение инструментов тестирования, изучение протокола 104 и Python.

---

## Текущее состояние

### Реализовано

| Компонент | Статус | Комментарий |
|-----------|--------|-------------|
| Сервер (listen, accept) | ✅ | Порт из config.toml, фильтр по `allow_ip` |
| U-кадры | ✅ | STARTDT/STOPDT/TESTFR — приём и ответ |
| Приём I-кадров | ✅ | Парсинг, обновление N(R) (rec_sq) |
| C_IC_NA_1 (общий опрос) | ⚠️ Частично | Отправляется только ACT_CON, **данные не выгружаются** |
| C_SC_NA_1 (однопозиционная ТУ) | ⚠️ Частично | ACT_CON отправляется, **значение не пишется в хранилище** |
| Отправка I-кадров | ✅ | M_SP_NA_1(1), M_SP_TB_1(30), M_ME_NC_1(13), M_ME_TF_1(36) |
| Хранилище сигналов | ✅ | DataStorage с подпиской клиентов, потокобезопасно |
| Конфиг | ✅ | TOML (сеть, протокол, лог), CSV сигналов (id, ca, ioa, asdu, name, val) |
| Имитация | ⚠️ Минимально | `imit.sim_float` + консольная команда `imit N id` |
| S-кадры | ❌ | `handle_s_frame` пустой; отправка S не реализована |
| Клиент 104 | ❌ | В репозитории — тестовый TCP-клиент (порт 3051), не протокол 104 |

### Пробелы для роли «симулятор для SCADA»

- **Общий опрос** — по C_IC_NA_1 нужно отправить все сигналы из хранилища с COT=INTERROGATED и завершить ACT_TERM; иначе WinCC OA и др. не получат полную картину после опроса.
- **Обратная связь по ТУ** — команда C_SC_NA_1 должна обновлять значение в хранилище (и при необходимости отражаться в последующих I-кадрах/общем опросе), иначе симулятор не имитирует «исполнение» команды.
- **S-кадры** — приём N(R) от клиента и периодическая отправка S для подтверждения принятых I-кадров (параметр w); без этого возможны разрывы и нестабильное поведение при интенсивном обмене.
- **Сценарии имитации** — воспроизводимые проверки SCADA требуют: сценарий по времени из CSV (время, id, значение) и/или циклическая имитация по группе сигналов с интервалом.
- **Документация и учебная ценность** — краткое описание 104 в контексте проекта, как подключать WinCC OA к симулятору, как запускать сценарии для проверок.

### Учебная ценность (уже есть / можно усилить)

- Изучение 104: разбор кадров в `protocol.py`, константы в `const.py`, формат CP56 в `protocol.py`.
- Python: threading, queue, dataclasses, конфиг (tomllib, csv).
- Инструмент тестирования: несколько экземпляров КП (KP_1, KP_2), один сервер на инстанс, сценарии для регрессий.

## Рекомендуемые приоритеты

1. **Общий опрос (C_IC_NA_1)** — выгрузка всех сигналов + ACT_TERM.
2. **Запись ТУ (C_SC_NA_1)** — обновление хранилища по IOA/значению.
3. **S-кадры** — приём N(R), отправка S по w.
4. **Имитация** — сценарий из CSV (время, id, значение); циклическая имитация по группе/интервалу.
5. **Документация** — цель проекта, структура, как тестировать SCADA (WinCC OA), примеры сценариев (в README или `docs/`).
6. **Опционально** — телерегулирование (C_SE_NC_1/C_SE_TC_1), двухпозиционная команда (C_DC_NA_1); клиент 104 для обратной проверки.

## Итог

Проект уже пригоден как заготовка симулятора и учебный стенд. Для уверенной проверки SCADA (WinCC OA и др.) критичны: полный ответ на общий опрос, запись ТУ в хранилище и поддержка S-кадров. Сценарии имитации и краткая документация доведут роль «симулятор полевого уровня + учебный проект» до целевого состояния.
